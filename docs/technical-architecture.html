Technical Architecture

Follow the journey of an authenticated photograph from camera capture through blockchain verification. Each step builds on the last to create an unforgeable chain of provenance.

1. On Camera

Everything begins at the moment you press the shutter. The camera creates cryptographic proof that this specific physical sensor captured this specific image.

The Hardware Root of Trust

Every camera sensor has microscopic manufacturing imperfections at the atomic level‚Äîno two sensors are identical. These imperfections create a unique "fingerprint" called a Non-Uniformity Correction (NUC) map that's as unique as your thumbprint.

At Manufacturing: The camera manufacturer captures this NUC fingerprint and stores a hash of it in the camera's Secure Element‚Äîa tamper-resistant chip that cannot be cloned or modified.

What Happens When You Take a Photo

The instant you press the shutter button, two parallel processes begin:

Process A: Image Hashing

Sensor captures light ‚Üí Raw Bayer data ‚Üì [Parallel split via DMA] ‚Üì ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î¥‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê ‚Üì ‚Üì Secure Element Image Signal Processor (ISP) ‚Üì Hash raw data (SHA-256) Process to JPEG/HEIC ‚Üì ‚Üì Raw Image Hash Processed Image ‚Üì (saved to memory) ‚Üì Hash processed image ‚Üì Processed Image Hash

Result: Two hashes‚Äîone of the raw sensor data (unmodified), one of the processed image (what you see).

Process B: Camera Token Generation

Secure Element retrieves stored NUC hash ‚Üì Randomly selects one of 3 assigned key tables ‚Üì Randomly selects a key from that table (1,000 options) ‚Üì Encrypts NUC hash with selected key (AES-256-GCM) ‚Üì Creates Camera Token: ‚Ä¢ Encrypted NUC hash (ciphertext) ‚Ä¢ Authentication tag (prevents tampering) ‚Ä¢ Nonce (unique encryption value) ‚Ä¢ Key reference (table ID + key index)

Result: Encrypted proof that only the manufacturer can verify‚Äîwithout seeing your image.

Performance: All of this happens in parallel with normal image processing‚Äîzero delay for the photographer.

Privacy by Default

  • GPS disabled: Location data is NOT hashed or transmitted by default
  • Capture time obscured: Only the server processing time is recorded, not when you took the photo
  • Image content private: Only hashes are created‚Äîthe image itself never leaves your camera unless you share it
  • Camera identity protected: The encryption key is shared by ~12,000 cameras, preventing individual tracking

2. Submission Packet

The camera bundles everything together into a submission packet and sends it to a Submission Server.

What's Inside the Packet

The camera creates a JSON packet containing three distinct pieces:

Part 1: Image Hashes

"image_hashes": [ { "image_hash": "a9d1dbb063ffd40ed3da020e...", // Raw sensor data "modification_level": 0, // Level 0 = Validated Raw "parent_image_hash": null }, { "image_hash": "29d6c8498815c58cb274cb48...", // Processed JPEG "modification_level": 1, // Level 1 = Validated "parent_image_hash": "a9d1dbb063ffd40ed3..." // Links to raw } ]

Why two hashes? The raw hash proves the sensor captured it. The processed hash lets you verify the actual image you share.

Part 2: Manufacturer Certificate

"manufacturer_cert": { "authority_id": "CANON_001", // Which manufacturer "validation_endpoint": "https://...", // Where to validate "camera_token": { "ciphertext": "2f358ac4a60fe672...", // Encrypted NUC hash "auth_tag": "6acb56f767fc4f1f...", // Tamper protection "nonce": "52428f0a..." // Unique encryption value }, "key_reference": { "table_id": 1203, // Which key table "key_index": 452 // Which key in table } }

Critical separation: The manufacturer certificate contains NO image hashes. The manufacturer will validate the camera without seeing what you photographed.

Part 3: Metadata

"submission_type": "camera", "timestamp": 1699564800 // When packet was created

Note: This timestamp is when the packet was assembled, NOT when the photo was taken‚Äîprotecting your privacy.

Where Does It Go?

The packet is transmitted to a Submission Server. You can choose which server to use:

  • Foundation-operated server: Run by the Birthmark Standard Foundation
  • Organization servers: News organizations may run their own (e.g., NYT, Reuters)
  • Your own server: Anyone can run a Submission Server‚Äîit's open-source

Security: Even if a server is compromised, the blockchain remains the source of truth. Servers are stateless relays.

3. Submission Server

The server acts as a router‚Äîseparating image hashes from manufacturer validation, then posting verified records to the blockchain.

The Separation Process

When a submission packet arrives, the server immediately splits it apart:

Routing Flow

Submission Packet Arrives ‚Üì [Extract manufacturer certificate] ‚Üì ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î¥‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê ‚Üì ‚Üì Send to Hold image hashes Manufacturer in queue Authority (NOT sent anywhere) ‚Üì Wait for PASS/FAIL ‚Üì ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î¥‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê ‚Üì ‚Üì PASS FAIL Post hashes Discard to blockchain submission

Privacy guarantee: The manufacturer never sees image hashes, timestamps, or which server received the submission. They only see the encrypted camera token.

What the Server Knows

‚úì Image hashes (needed to post to blockchain)

‚úì Key table/index reference (not the actual camera)

‚úì Manufacturer validation result (PASS or FAIL)

‚úó Which specific camera (~12,000 cameras share each key)

‚úó Camera's NUC hash (encrypted, can't decrypt)

‚úó Image content (never transmitted)

Multiple Servers, One Blockchain

Any number of Submission Servers can operate simultaneously, all posting to the same blockchain. This means:

  • No single point of failure: If one server goes down, others continue
  • Competition: Servers compete on performance and privacy policies
  • Open participation: Anyone can run a server‚Äîno permission required
  • Unified verification: All records end up on the same blockchain regardless of server

4. Manufacturer Authority

The manufacturer validates that a legitimate camera created the token‚Äîwithout learning anything about the photograph itself.

The Validation Process

What the Manufacturer Receives

POST /validate { "camera_token": { "ciphertext": "2f358ac4a60fe672...", "auth_tag": "6acb56f767fc4f1f...", "nonce": "52428f0a..." }, "key_reference": { "table_id": 1203, "key_index": 452 } }

That's it. No image hashes, no timestamps, no server identity‚Äîjust the encrypted token and key reference.

Manufacturer Validation Steps

1. Look up decryption key from key database (Table 1203, Key 452) 2. Decrypt the camera token using AES-256-GCM ‚Üì Decrypted NUC hash 3. Compare against production database "Is this NUC hash from one of our cameras?" 4. Check authentication tag "Has this token been tampered with?" 5. Return response: PASS - "Yes, one of our cameras" FAIL - "Unknown camera or tampered token"

Privacy Architecture

Key Table Anonymity: Each camera is provisioned with 3 random key tables at manufacturing. Each table contains 1,000 keys and is shared by approximately 12,000 cameras.

What this means:

  • Manufacturer learns "one of ~12,000 cameras authenticated a photo"
  • Cannot determine which specific camera without correlating many submissions
  • Cannot track individual photographers or link photos together
  • Camera randomly rotates between 3 tables, further obscuring patterns

What Manufacturer Sees

‚úì "One of my cameras validated a photo"

‚úì Usage statistics (how many validations per day)

‚úó Which specific camera (anonymity set of ~12,000)

‚úó Image content or hashes

‚úó When photo was taken

‚úó Where photo was taken

‚úó Which Submission Server processed it

Trust Model

The manufacturer authority is trusted to:

  • ‚úì Maintain accurate production records
  • ‚úì Properly validate camera tokens
  • ‚úì Protect key table databases from compromise

The manufacturer authority is not trusted with:

  • ‚úó Image content (never receives it)
  • ‚úó Photographer privacy (anonymity by design)
  • ‚úó Blockchain decisions (coalition-controlled)

5. Blockchain (Birthmark Media Registry)

The permanent, public record. Once a hash is recorded here, anyone can verify it‚Äîforever.

Why Blockchain?

The fundamental problem: C2PA and similar solutions embed authentication metadata in image files. When you share an image on social media, upload to Imgur, or email it, that metadata is stripped. 95% of images lose their authentication.

The Birthmark solution: Store authentication records on an independent blockchain. The image file can be compressed, converted, cropped‚Äîthe hash remains verifiable by querying the blockchain.

Record Structure

Each blockchain entry contains minimal, essential information:

{ "image_hash": "a9d1dbb063ffd40ed3da020e14aa994a...", // SHA-256 hash "submission_type": "camera", // or "software" "modification_level": 0, // 0=Raw, 1=Validated, 2=Modified "modification_display": "Validated Raw", "parent_image_hash": null, // or parent hash if edited "authority_id": "CANON_001", // Which manufacturer "submission_server_id": "server_public_key", "timestamp": 1699564800, // Server processing time "block_number": 12345 }

Storage size: ~140 bytes per record (optimized from original 450 bytes)

Coalition Governance

The blockchain is operated by the Media Validation Network‚Äîa coalition of mission-aligned organizations:

  • Universities: Academic institutions with long-term stability
  • Archives: Organizations dedicated to preservation (Internet Archive, university libraries)
  • Journalism organizations: Press freedom and fact-checking groups
  • Foundation: The Birthmark Standard Foundation (technology developer and voting member)

Explicitly excluded: Media production companies, camera manufacturers, tech platforms‚Äîto avoid conflicts of interest.

Consensus: Proof of Authority with supermajority (67%) required for blockchain changes. This means 7 of 10 organizations would need to collude to attack the system‚Äîeach with significant reputational stakes.

Economic Sustainability

Operating a validator node is designed to be affordable for public institutions:

  • 1M images/day: ~47 GB/year storage, $200-350/year node cost
  • 10M images/day: Scales to $500-800/year per node
  • Zero transaction fees: Public service model, no gas fees
  • Standard hardware: Runs on commodity servers

Result: Universities and archives can operate nodes as a public service‚Äîlike hosting open-access journals.

Data Availability

Key innovation: Individual image hashes are stored directly on-chain, not in external databases. This means:

  • If a Submission Server disappears, records remain accessible
  • No dependency on centralized databases
  • Verification works even if 90% of servers fail
  • Anyone can run a full node and verify independently

6. Verifier (Public Verification)

Anyone can verify any image‚Äîno account required, no subscription, no gatekeepers.

How Verification Works

Verification Flow

User has an image (JPEG, PNG, HEIC, etc.) ‚Üì Upload to web verifier or hash locally ‚Üì Calculate SHA-256 hash of image bytes ‚Üì Query blockchain: GET /verify/{image_hash} ‚Üì Blockchain returns record (if exists): ‚Ä¢ Modification level (Validated Raw / Validated / Modified) ‚Ä¢ Authority ID (which manufacturer/developer) ‚Ä¢ Timestamp (when authenticated) ‚Ä¢ Parent hash (if edited) ‚Ä¢ Provenance chain (full edit history) ‚Üì Display result to user

Verification Results

Modification Level 0: "Validated Raw"

Meaning: This is raw sensor data from a verified camera‚Äîno processing beyond sensor readout.

Use case: Maximum authenticity for evidentiary purposes. Typically seen only by forensic analysts.

Color: Green

Modification Level 1: "Validated"

Meaning: This is a processed image (JPEG/HEIC) from a verified camera. Standard white balance, sharpening, and compression applied by the camera.

Use case: Standard authentication level for photojournalism and social sharing.

Color: Green

Modification Level 2: "Modified"

Meaning: This image was edited in software (Lightroom, Photoshop, etc.) but links back to a validated original.

Use case: Edited photos that maintain chain of custody to original capture.

Color: Yellow/Orange

Not Found

Meaning: No record found. Either never authenticated, or blockchain doesn't have this hash yet.

Use case: Unknown provenance. Treat with appropriate skepticism.

Color: Gray

Provenance Chains

When an image is edited through validated software, the verifier shows the full history:

üì∏ Original (Level 1: Validated) Hash: a9d1dbb063ffd40e... Authority: CANON_001 Date: 2024-03-15 ‚Üì [Edited in Lightroom] ‚Üì üé® Edited (Level 2: Modified) Hash: 3f7a8b2c1d9e4f5a... Authority: ADOBE_LIGHTROOM Parent: a9d1dbb063ffd40e... Date: 2024-03-15

Transparency: Viewers know the image was edited, what software was used, and can trace back to the original camera capture.

Open Verification Protocol

The verification API is completely open:

  • No API key required: Anyone can query the blockchain
  • No rate limits: Public good, unlimited access
  • Multiple implementations: Web app, browser extension, command-line tool, mobile apps
  • Anyone can build: Verification tools, fact-checking integrations, newsroom plugins

Example integrations:

  • Twitter/X bot: Reply to images with verification status
  • Newsroom CMS plugins: Auto-verify submitted photos
  • Fact-checking tools: Batch verification for investigations
  • Browser extensions: Right-click any image to verify

7. Editor Pipeline

Professional photographers need to edit their work. The Birthmark Standard supports editing while maintaining provenance and transparency.

The Editing Challenge

Problem: Professional photography requires editing‚Äîcolor correction, cropping, exposure adjustment. But editing changes the image hash, breaking traditional authentication.

Naive solution: Only allow unedited images ‚Üí unacceptable for professional workflows.

Birthmark solution: Re-authenticate edited images through validated editing software, maintaining a chain of custody back to the original camera capture.

How Editor Reauthentication Works

Step-by-Step Process

1. Photographer takes photo with Birthmark camera ‚Üí Image hash posted to blockchain ‚Üí Modification Level 1: "Validated" 2. Import image into validated editing software (e.g., Adobe Lightroom, Capture One) 3. Make edits (crop, color correction, etc.) 4. Export edited image ‚Üí Editor creates NEW submission packet: ‚Ä¢ Hash of edited image ‚Ä¢ Hash of parent (original) image ‚Ä¢ Developer certificate (proves legitimate software) ‚Ä¢ Edit manifest (what operations were performed) 5. Submit to Submission Server ‚Üí Routes to Developer Authority (not manufacturer) ‚Üí Developer validates software authenticity ‚Üí If PASS: post to blockchain 6. Blockchain records edited image with: ‚Üí Modification Level 2: "Modified" ‚Üí Parent hash linking to original ‚Üí Authority: ADOBE_LIGHTROOM (or other editor)

Developer Authority

Parallel to the Manufacturer Authority, but for software:

What Developer Authority Validates

  • Software authenticity: Is this legitimate Lightroom/Photoshop/etc.?
  • License status: Is this a licensed copy?
  • Edit manifest integrity: Has the edit record been tampered with?

Privacy preserved: Developer never sees image content or hashes‚Äîonly software validation token.

Implementation options:

  • Option 1: Software developer operates authority (e.g., Adobe validates Lightroom)
  • Option 2: Third-party authority validates multiple editors (e.g., Foundation validates open-source tools)
  • Option 3: Self-authentication for open-source tools (cryptographic signing)

Edit Manifest

Optionally, editing software can include a manifest of operations performed:

{ "edit_operations": [ {"operation": "crop", "params": "16:9 aspect ratio"}, {"operation": "exposure", "params": "+0.5 EV"}, {"operation": "color_temperature", "params": "5500K ‚Üí 6000K"}, {"operation": "saturation", "params": "+10%"} ], "software_version": "Adobe Lightroom Classic 13.1", "timestamp": 1699564800 }

Use case: Fact-checkers and editors can see exactly what edits were applied, distinguishing routine adjustments from substantive manipulations.

Privacy: Edit manifest is optional. Photographers can choose to include it or submit only the hash linkage.

Multi-Generation Editing

Images can be edited multiple times, creating chains:

üì∏ Camera Capture (Level 1) ‚Üì üé® Lightroom Edit (Level 2) ‚Üì üé® Photoshop Edit (Level 2) ‚Üì üé® Final Web Export (Level 2)

Each step:

  • Links to its parent via hash
  • Validated by Developer Authority
  • Recorded on blockchain
  • Publicly verifiable

Result: Complete provenance transparency‚Äîviewers see the full editing history from camera capture to final image.

Acceptable Edit Types

The system supports standard photographic editing while flagging manipulations:

Standard Edits (Modification Level 2)

  • Exposure, contrast, brightness adjustments
  • Color correction, white balance
  • Cropping and rotation
  • Sharpening and noise reduction
  • Lens correction
  • Format conversion (RAW ‚Üí JPEG)

Substantive Manipulations (Flagged in Edit Manifest)

  • Content-aware fill / object removal
  • Sky replacement
  • Face/body modifications
  • Composite/blended images
  • AI enhancement tools

Note: These operations are allowed and recorded, but clearly flagged so viewers understand the image was substantially altered.

Adoption Strategy

Phase 1: Foundation-validated open-source editors (GIMP, Darktable)

Phase 2: Partner with major software companies (Adobe, Capture One, Affinity)

Phase 3: Mobile editing apps (Snapseed, VSCO, Darkroom)

Incentive for developers: Authenticated editing becomes a feature advantage. "Edit without breaking authentication" is a compelling selling point for professional photographers.

Integration with Existing Standards

Relationship to C2PA

Complementary, Not Competitive:

  • C2PA: Rich metadata, existing tool ecosystem, but lost when metadata stripped
  • Birthmark: Blockchain fail-safe that survives metadata loss
  • Together: Comprehensive authentication covering all distribution scenarios

Key Differentiators:

  • C2PA validates "something with this certificate signed this"
  • Birthmark validates "this specific physical camera sensor captured this"
  • C2PA vulnerable to firmware exploits (Nikon Z6 III incident)
  • Birthmark requires physical sensor access to forge

"Implement Birthmark alongside C2PA for comprehensive authentication. C2PA handles rich metadata and existing workflows. Birthmark handles the 95% of images that lose metadata through social media sharing."

Security Considerations

Threat Mitigations

Threat: Camera Token Forgery

Risk: Attacker extracts NUC hash and creates fake tokens

Protection: NUC maps physically unforgeable (atomic-scale manufacturing variance). Attack cost: $100K-1M per camera (requires destructive hardware analysis + secure element extraction). Manufacturer validates against production records.

Threat: Submission Server Compromise

Risk: Server manipulates submissions or goes offline

Mitigation: All data on blockchain, server is stateless relay. Spin up new server, continues from blockchain state. Multiple servers can operate simultaneously.

Threat: Coalition Member Collusion

Risk: Blockchain validators collude to manipulate records

Protection: Supermajority (67%) needed for malicious action‚Äîrequires 7 of 10 organizations to collude. Coalition members have reputational stakes in integrity. Media producers excluded, reducing conflicts of interest.

Threat: Government Subversion

Risk: Government compels coalition members to reveal information

Protection: Legal compulsion triggers removal, not compliance. Organization exits coalition rather than compromising. Network continues operating with remaining nodes.

What the System Does NOT Protect Against

  • Staged scenes: Real camera capturing fake/staged scene
  • Images of screens: Camera authenticates the screen, not original
  • Social engineering: Tricking coalition members
  • Side-channel attacks: Timing analysis, power monitoring (mitigated by constant-time crypto)

Scope: The system proves "this specific camera captured this," not "this image represents reality." Context and journalistic judgment remain essential.
